Now I understand the issue. When the domain contains non-ASCII characters (like `münchen`), it tries to encode the whole domain with `idna`. But the issue is that `münchen` encodes to `xn--mnchen-3ya` which doesn't start or end with a hyphen, but `münchen-` would encode to something ending with a hyphen, and `-münchen` would encode to something starting with a hyphen.

Wait, actually the issue is different. The `idna` encoding of the whole domain might succeed even when individual labels start or end with hyphens. Let me think about this more carefully.

The problem is that `user@-münchen.de` - the label `-münchen` when encoded via IDNA becomes `xn---mnchen-n2a` or similar, which starts with `xn--` but the underlying label starts with a hyphen. The DOMAIN_REGEX checks `[A-Z0-9](?:[A-Z0-9-]{0,61}[A-Z0-9])?` which requires labels to start and end with alphanumeric, but after IDNA encoding the `xn--` prefix might mask the issue.

The fix should encode each label individually and check if the original label starts/ends with a hyphen.
