Looking at the code, the issue is in the `Email.__call__` method. When the domain part fails the `DOMAIN_REGEX` match, it tries to encode it with IDNA encoding. However, the IDNA encoding itself doesn't reject labels starting or ending with hyphens - that check needs to happen after encoding.

The fix should validate that after IDNA encoding, the resulting domain labels don't start or end with hyphens. Actually, the real issue is that `domain_part.encode("idna")` may succeed even for labels starting/ending with hyphens in some Python versions, or the DOMAIN_REGEX should catch it after encoding.

Let me think: `DOMAIN_REGEX` is `r"(?:[A-Z0-9](?:[A-Z0-9-]{0,61}[A-Z0-9])?\.)+"` - this requires labels to start and end with `[A-Z0-9]`, so it should reject labels starting/ending with hyphens. The problem is that after IDNA encoding, `münchen` becomes `xn--mnchen-3ya` (valid), but `-münchen` becomes `xn---mnchen-3ya` or similar which might still match or the encoding might succeed and produce something that matches.

The fix is to check each label of the original domain for leading/trailing hyphens before attempting IDNA encoding.
